  defmodule Accrue.Telemetry.Ops do
    @moduledoc """
    Ops-grade telemetry emit helper (OBS-03 / D4-02 / D4-04).

    High-signal events go under `[:accrue, :ops, :*]`. SREs subscribe to this namespace
    for alertable conditions. The firehose `[:accrue, :*]` namespace contains every
    public entry point and is too noisy for alerts.

    Canonical ops events in Phase 4:
      [:accrue, :ops, :revenue_loss]
      [:accrue, :ops, :dunning_exhaustion]
      [:accrue, :ops, :incomplete_expired]
      [:accrue, :ops, :charge_failed]
      [:accrue, :ops, :meter_reporting_failed]
      [:accrue, :ops, :webhook_dlq, :dead_lettered]
      [:accrue, :ops, :webhook_dlq, :replay]
      [:accrue, :ops, :webhook_dlq, :prune]
    """

    @spec emit(atom() | [atom()], map(), map()) :: :ok
    def emit(suffix, measurements, metadata \\ %{})

    def emit(suffix, measurements, metadata) when is_atom(suffix) do
      emit([suffix], measurements, metadata)
    end

    def emit(suffix, measurements, metadata) when is_list(suffix) and is_map(measurements) and is_map(metadata) do
      event = [:accrue, :ops] ++ suffix
      merged_metadata =
        metadata
        |> Map.put_new_lazy(:operation_id, fn -> Accrue.Context.operation_id() end)

      :telemetry.execute(event, measurements, merged_metadata)
      :ok
    end
  end
  ```

  ```markdown
  # Telemetry & Observability

  Accrue emits `:telemetry` events for every public entry point. This guide documents
  the conventions, the high-signal `[:accrue, :ops, :*]` namespace, and the default
  `Telemetry.Metrics` recipe.

  ## Namespace split

  Accrue divides telemetry into two namespaces:

  - **`[:accrue, :*]` — the firehose.** Every public Accrue entry point emits
    `:start`/`:stop`/`:exception` here. High cardinality. Use for tracing and debug,
    NOT for paging SREs.
  - **`[:accrue, :ops, :*]` — SRE-actionable signals.** Low cardinality, high value.
    Every event here is an alertable condition on someone's on-call rotation.

  ## Ops events in v1.0

  | Event | Measurements | Metadata |
  |-------|-------------|----------|
  | `[:accrue, :ops, :revenue_loss]` | `count`, `amount_minor`, `currency` | `subject_type`, `subject_id`, `reason` |
  | `[:accrue, :ops, :dunning_exhaustion]` | `count` | `subscription_id`, `from_status`, `to_status`, `source` (`:accrue_sweeper \| :stripe_native \| :manual`) |
  | `[:accrue, :ops, :incomplete_expired]` | `count` | `subscription_id` |
  | `[:accrue, :ops, :charge_failed]` | `count` | `charge_id`, `customer_id`, `failure_code` |
  | `[:accrue, :ops, :meter_reporting_failed]` | `count` | `meter_event_id`, `event_name`, `source` (`:reconciler \| :webhook \| :sync`) |
  | `[:accrue, :ops, :webhook_dlq, :dead_lettered]` | `count` | `event_id`, `processor_event_id`, `type`, `attempt` |
  | `[:accrue, :ops, :webhook_dlq, :replay]` | `count`, `duration`, `requeued_count`, `skipped_count` | `actor`, `filter`, `dry_run?` |
  | `[:accrue, :ops, :webhook_dlq, :prune]` | `dead_deleted`, `succeeded_deleted`, `duration` | `retention_days` |

  All ops events fire inside the same `Repo.transact/2` as the state write they
  correspond to — they are idempotent under webhook replay via the `accrue_webhook_events`
  unique index short-circuit.

  ## Span naming conventions (OpenTelemetry)

  Accrue's OpenTelemetry span helpers (gated on `:opentelemetry` optional dep) wrap
  every Billing context function with consistent naming:

  ```
  accrue.<domain>.<resource>.<action>
  ```

  Examples:
  - `accrue.billing.subscription.create`
  - `accrue.billing.invoice.finalize`
  - `accrue.webhooks.dlq.replay`
  - `accrue.checkout.session.create`
  - `accrue.billing_portal.session.create`

  Span kind:
  - `INTERNAL` for Accrue context functions
  - `CLIENT` for outbound `lattice_stripe` calls

  ## Attribute conventions

  **Allowed attributes** (host-queryable, audit-useful):
  - `accrue.subscription.id` — Accrue's internal UUID
  - `accrue.customer.id`
  - `accrue.invoice.id`
  - `accrue.event_type` — webhook event type string
  - `accrue.processor` — `:stripe` | `:fake`
  - `stripe.subscription.id` — upstream Stripe ID (for support bridging)
  - `stripe.customer.id`
  - `stripe.charge.id`

  **PROHIBITED attributes** (never attach to spans/events):
  - Any customer email, name, phone, address
  - Any card PAN, CVC, expiry
  - Any Stripe `PaymentMethod` raw data
  - Any webhook raw body
  - Any `Accrue.Money` amounts that identify a specific customer purchase history

  Attach PII-free identifiers ONLY. If an attribute could be reversed into a person,
  don't attach it.

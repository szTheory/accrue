<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local shell -> Phoenix host app | Script-controlled setup and server boot exercise the real host boundary |
| CI runtime -> GitHub API | Workflow metadata and annotations are fetched through a tokened API boundary |
| CI logs/artifacts -> developers | Server logs and annotation summaries become visible to humans outside the runner |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-02-01 | Information Disclosure | host UAT logs and browser logs | mitigate | Print only bounded server logs on failure and do not echo Stripe secrets, webhook secrets, cookies, or fixture passwords. |
| T-11-02-02 | Tampering | generated-drift and compile/test order | mitigate | Keep the install rerun, drift check, warning-as-error compile, and host test stages before browser success can be reported. |
| T-11-02-03 | Repudiation | annotation blocker | mitigate | Make annotation sweep deterministic from `GITHUB_REPOSITORY` + `GITHUB_RUN_ID` and emit the exact job/annotation summary that triggered failure. |
| T-11-02-04 | Elevation of Privilege | GitHub token use in sweep script | mitigate | Use the default workflow token only for read-only API queries, do not request write scopes in the script contract, and fail closed when auth is missing. |
| T-11-02-05 | False Negative | annotation warnings survive without failing CI | mitigate | Add a dedicated sweep script that fails on surviving `warning` or `failure` annotations for named release-facing jobs. |
</threat_model>

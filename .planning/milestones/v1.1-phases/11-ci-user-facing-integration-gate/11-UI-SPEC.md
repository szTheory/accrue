# Phase 11 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Existing Phoenix LiveView host UI plus the shipped `accrue_admin` component system; do not introduce a second browser test surface or alternate UI shell in CI |
| Icon library | Existing Heroicons in the host app and text-first `accrue_admin` controls; browser assertions should prefer accessible names over icon-only hooks |
| Font | `system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif` |

Phase 11 inherits Phase 10's user-facing copy, action labels, and state transitions. This phase does not redesign the host app. It turns the existing host and admin browser path into a release-blocking CI contract with retained failure evidence.

---

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Tight inline gaps, badge dots, compact metadata spacing |
| sm | 8px | Button/icon gaps, flash internals, compact action spacing |
| md | 16px | Default form, flash, and card rhythm in the host/browser proof path |
| lg | 24px | Section padding, grouped confirmation controls, panel breathing room |
| xl | 32px | Page gutters and major block spacing in host billing and admin detail pages |
| 2xl | 48px | Empty-state breathing room and large section separation |
| 3xl | 64px | Page-level vertical separation only; no new oversized promo spacing |

Exceptions: 44px minimum target for clickable controls and disclosure triggers. Browser checks should fail on clipped or overlapping text rather than adding one-off spacing tokens.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Typography remains inherited from the Phase 10 host flow and the existing `accrue_admin` shell. CI should validate readable, non-truncated states rather than pixel-snapshot typography diffs.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FAFBFC` light / `#0F1318` dark | App background, host shell, admin shell, report screenshots of normal page state |
| Secondary (30%) | `#FFFFFF` light / `#171D24` dark | Cards, confirmation panels, flash containers, admin elevated surfaces |
| Accent (10%) | `#5D79F6` | Primary subscription CTA, selected admin nav/tab state, actionable links, focus ring, retained success-path emphasis |
| Destructive | `#111418` | Confirm-cancel and replay/refund confirmation paths only; never a generic accent substitute |

Accent reserved for: `Start subscription`, authenticated navigation into billing, active admin navigation/tab state, focus-visible outlines, and the confirmed action surface that follows Phase 10's host/admin flow. Do not use accent as a blanket fill for every badge, card, or failure state.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Start subscription |
| Empty state heading | No billing activity yet |
| Empty state body | Billing records appear after a user starts a subscription or a webhook is processed. Start a subscription or review the webhook feed. |
| Error state | We couldn't complete that billing action. Check auth, migrations, webhook signing, or processor setup, then try again. |
| Destructive confirmation | Cancel subscription: Confirm cancellation before ending access. Replay webhook: Webhook replay requested. Refund flow remains out of scope for the Phase 11 host gate unless already exercised by the inherited admin browser suite. |

Browser assertions must use these exact visible strings where they already exist in the host app or mounted admin UI. Do not replace them with test-only copy.

## Browser UAT Contract

| Area | Contract |
|------|----------|
| Primary visual hierarchy | The host billing view's first focal point is the current subscription or empty billing state, followed by the primary `Start subscription` action and then supporting projection/event context. The admin detail flow's first focal point is the selected webhook/event identity and status, followed by replay controls and audit/event evidence. |
| Required flow coverage | One blocking browser path must cover signed-in host user billing entry, empty state, plan start, cancellation confirmation, admin entry, webhook detail, replay action, and resulting audit/event evidence. |
| Viewport coverage | Required blocking project: Chromium desktop at `1280x900`. Reuse existing `accrue_admin` multi-project coverage if already present, but Phase 11 must not block on inventing new mobile-only host assertions. |
| Artifact retention | On browser failure, upload Playwright HTML report, Playwright trace output, failure screenshots, and host Phoenix server log. Retain artifacts only on failure to keep the required gate legible. |
| Report shape | Use Playwright Test with the GitHub reporter plus HTML report in CI. The browser gate must not remain on a raw `node` smoke script once Phase 11 lands. |
| Failure evidence quality | Every failed run must answer four questions without a local rerun: which step failed, which visible state was expected, what the page looked like, and what the server logged during the failure window. |
| Assertion style | Prefer `getByRole`, `getByLabel`, `getByText`, `aria-*`, and URL/state assertions. Use `data-role` or `data-plan-id` only for stable action hooks where semantics are not otherwise exposed. |
| Accessibility state checks | Assert accessible landmarks or names already present in the inherited UI: `Choose a plan`, `Current subscription`, `Local billing projections at a glance`, `invoice.payment_failed`, `Admin navigation`, and success/error flash copy. |
| LiveView readiness | Wait for the mounted LiveView to connect before asserting post-mount state. Treat disconnected or half-hydrated DOM as a failure artifact case, not a flaky retry surface. |
| Visual consistency boundary | CI validates the actual Phase 10 host/admin UX. Do not add a separate test fixture UI, alternate route, hidden dev chrome, or CI-only styles just to simplify browser checks. |
| Non-goals | Exact pixel snapshots, theme restyling, marketing pages, and new responsive redesign work are out of scope for Phase 11. Phase 14 owns broad responsive and accessibility hardening. |

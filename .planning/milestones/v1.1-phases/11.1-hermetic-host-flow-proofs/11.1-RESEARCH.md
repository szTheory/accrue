## Summary

The current gap is real and narrow. A fresh migrated `examples/accrue_host` test database passes the direct focused proof command, and the canonical `bash scripts/ci/accrue_host_uat.sh` wrapper also passes. [VERIFIED: local repro] The failure appears only after the wrapper's Playwright phase leaves committed `MIX_ENV=test` rows behind and the next direct focused proof run restarts `Accrue.Processor.Fake` at `sub_fake_00001`, causing `accrue_subscriptions_processor_processor_id_index` collisions plus stale webhook counts. [VERIFIED: local repro][VERIFIED: `accrue/lib/accrue/processor/fake.ex`][VERIFIED: `examples/accrue_host/test/accrue_host_web/*.exs`]

The standard fix is not to randomize the Fake processor or to broaden wrapper-level database resets. [VERIFIED: `accrue/lib/accrue/processor/fake.ex`][VERIFIED: `.planning/v1.1-MILESTONE-AUDIT.md`] The right fix for this phase is a proof-specific setup layer that runs before each focused proof, resets `Accrue.Processor.Fake`, and deletes the exact persisted host-flow records that the browser path or prior direct runs can leave behind, while keeping realistic `price_basic`, signed webhook, and admin replay evidence inside each test body. [VERIFIED: local repro][VERIFIED: `accrue/test/support/billing_case.ex`][CITED: https://hexdocs.pm/ex_unit/ExUnit.Callbacks.html]

**Primary recommendation:** Add a dedicated host-flow proof case/helper that combines sandbox ownership, `Accrue.Processor.Fake.reset/0`, and targeted cleanup of committed host-flow rows, then make the three focused proof files use it and verify both the direct three-file command and `scripts/ci/accrue_host_uat.sh`. [VERIFIED: local repro][VERIFIED: `accrue/test/support/billing_case.ex`]

## Architecture Patterns

### System Architecture Diagram

```text
direct mix test command
  -> ExUnit setup in proof file / case template
    -> sandbox owner + Fake.reset + targeted cleanup
      -> host proof action
        -> AccrueHost.Billing / router / admin LiveView
          -> Accrue.Billing + Accrue.Webhook + Accrue.Processor.Fake
            -> Postgres tables (users, accrue_customers, accrue_subscriptions, accrue_webhook_events, accrue_events, oban_jobs)
              -> proof assertions

wrapper command
  -> scripts/ci/accrue_host_uat.sh
    -> compile + focused tests + full suite + browser seed + test Phoenix server + Playwright
      -> committed MIX_ENV=test browser actions
        -> same Postgres tables
          -> later direct focused rerun must still pass
```

The reproduced gap sits between the wrapper's committed browser path and the later direct focused rerun. [VERIFIED: local repro]

### Pattern 1: Proof-Local State Reset Plus Targeted Cleanup
**What:** Before each focused proof, start or reuse `Accrue.Processor.Fake`, call `Fake.reset/0`, and delete the exact persisted rows that prior browser/direct runs can leave behind. [VERIFIED: local repro][VERIFIED: `accrue/test/support/billing_case.ex`]
**When to use:** For the three focused host-flow files that must pass after prior committed `MIX_ENV=test` activity. [VERIFIED: `.planning/v1.1-MILESTONE-AUDIT.md`][VERIFIED: local repro]
**Example:**
```elixir
# Source: local case-template pattern + ExUnit callbacks docs
defmodule AccrueHost.HostFlowProofCase do
  use ExUnit.CaseTemplate

  using do
    quote do
      use AccrueHostWeb.ConnCase, async: false
      import AccrueHost.HostFlowProofCase
    end
  end

  setup context do
    case Accrue.Processor.Fake.start_link([]) do
      {:ok, _} -> :ok
      {:error, {:already_started, _}} -> :ok
    end

    :ok = Accrue.Processor.Fake.reset()
    cleanup_host_flow_rows!()
    {:ok, context}
  end
end
```

This matches the existing core test-case pattern that resets Fake per test. [VERIFIED: `accrue/test/support/billing_case.ex`][CITED: https://hexdocs.pm/ex_unit/ExUnit.Callbacks.html]

### Pattern 2: Deterministic Business Inputs, Unique or Cleaned Persistence Keys
**What:** Keep realistic stable business inputs such as `price_basic`, route paths, signed payload shape, and admin audit types, but make persisted ids unique where tests manually seed rows or delete conflicting rows before using deterministic Fake ids again. [VERIFIED: proof files][VERIFIED: local repro]
**When to use:** For `processor_event_id`, manually inserted replay fixtures, and browser seed helpers. [VERIFIED: `examples/accrue_host/test/accrue_host_web/webhook_ingest_test.exs`][VERIFIED: `examples/accrue_host/test/accrue_host_web/admin_webhook_replay_test.exs`][VERIFIED: `scripts/ci/accrue_host_seed_e2e.exs`]
**Example:**
```elixir
# Source: existing repo practice of unique suffixes in tests
processor_event_id = "evt_host_replay_" <> Integer.to_string(System.unique_integer([:positive]))
```

The repo already uses `System.unique_integer/1` broadly for persisted fixture ids where deterministic global ids are not required. [VERIFIED: code grep]

### Anti-Patterns to Avoid
- **Reset-only isolation:** Calling `Accrue.Processor.Fake.reset/0` without deleting committed rows is insufficient and reproduces the unique-index collision. [VERIFIED: local repro]
- **Whole-database teardown inside proof setup:** Dropping and recreating the database inside each focused proof would hide the bug, slow local iteration, and duplicate the wrapper's job. [VERIFIED: current wrapper responsibilities][ASSUMED]
- **Changing Fake's global id semantics for one host phase:** The Fake is intentionally deterministic across the project, and Phase 11.1 does not need a core-processor redesign. [VERIFIED: `accrue/lib/accrue/processor/fake.ex`][VERIFIED: `.planning/REQUIREMENTS.md`]

## Common Pitfalls

### Pitfall 1: Confusing Sandbox Rollback With Full-System Hermeticity
**What goes wrong:** The ExUnit files look isolated because they use `ConnCase`, but later direct reruns still fail after the browser wrapper has run. [VERIFIED: local repro]
**Why it happens:** `ConnCase` protects the proof transaction, while `MIX_ENV=test mix phx.server` plus Playwright runs outside that sandbox and commits rows. [VERIFIED: `examples/accrue_host/test/support/conn_case.ex`][VERIFIED: `scripts/ci/accrue_host_uat.sh`]  
**How to avoid:** Clean persisted host-flow rows in proof setup or namespaced seed helpers, and reserve wrapper-level `ecto.drop/create/migrate` for full-suite orchestration only. [VERIFIED: local repro]
**Warning signs:** `accrue_subscriptions_processor_processor_id_index` collisions on `sub_fake_00001` or stale `WebhookEvent` counts before the test inserts anything. [VERIFIED: local repro]

### Pitfall 2: Cleaning Only Subscriptions
**What goes wrong:** Subscription collisions disappear, but the webhook tamper test or admin replay assertions still see leftover webhook and event rows. [VERIFIED: local repro]
**Why it happens:** The direct proof set touches `users`, `accrue_customers`, `accrue_subscriptions`, `accrue_subscription_items`, `accrue_webhook_events`, `accrue_events`, and `oban_jobs`. [VERIFIED: proof files][VERIFIED: `scripts/ci/accrue_host_seed_e2e.exs`]
**How to avoid:** Define one cleanup helper that deletes the full host-flow footprint in FK-safe order or scopes by stable prefixes/emails. [VERIFIED: proof files][ASSUMED]
**Warning signs:** The tampered webhook test expects zero webhook rows and instead sees one existing row before its POST. [VERIFIED: local repro]

### Pitfall 3: Solving the Wrong Path
**What goes wrong:** The implementation starts plumbing full acceptance-test sandbox metadata into Phoenix before proving the direct three-file rerun contract. [CITED: https://hexdocs.pm/phoenix_ecto/Phoenix.Ecto.SQL.Sandbox.html][VERIFIED: local repro]
**Why it happens:** `Phoenix.Ecto.SQL.Sandbox` is the official answer for concurrent browser tests, but the reproduced audit gap is simpler. [CITED: https://hexdocs.pm/phoenix_ecto/Phoenix.Ecto.SQL.Sandbox.html][VERIFIED: local repro]
**How to avoid:** First close the gap with proof-local cleanup and unique manual fixture ids; only escalate to sandbox metadata if the planner deliberately broadens Phase 11.1 into concurrent acceptance coverage. [VERIFIED: local repro]
**Warning signs:** Endpoint/socket/header changes appear in the plan even though the failing command is still `mix test ... subscription_flow_test.exs webhook_ingest_test.exs admin_webhook_replay_test.exs`. [VERIFIED: milestone audit][VERIFIED: local repro]

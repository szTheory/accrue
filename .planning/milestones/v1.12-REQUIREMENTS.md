# Requirements: Accrue v1.12

**Defined:** 2026-04-22  
**Milestone:** v1.12 — Admin & operator UX  
**Core value:** A Phoenix developer can install Accrue + its companion admin UI, and launch a real SaaS with subscription billing on day one — complete, production-grade, idiomatic, with tamper-evident audit ledger, great observability, and zero breaking-change pain through v1.x.

Milestone theme: **`accrue_admin`** operator surfaces (post-metering signals, drill/nav, **`AccrueAdmin.Copy`** + token discipline, VERIFY-01 extensions)—**not** **PROC-08**, **FIN-03**, new third-party UI kits, or core billing schema changes.

---

## Admin & operator surfaces (ADM)

- [x] **ADM-01**: Operators landing on the **default admin home** (or the phase-scoped entry surface) see at least one **credible metering- or usage-adjacent signal** (counts, recent failures, or deep link into an existing index such as webhooks/meter-related surfaces) consistent with shipped **v1.10** semantics and existing **telemetry / runbook** narratives — **no** new accounting semantics. *(Validated Phase 48 — plan 01, 2026-04-22.)*
- [x] **ADM-02**: At least **one** high-traffic **list or detail** flow (e.g. customer ↔ subscription ↔ invoice) is measurably smoother: fewer dead ends, clearer next actions, or preserved context — scoped in the phase plan to **concrete LiveView routes**. *(Validated Phase 49 — plans 01–02, 2026-04-22.)*
- [x] **ADM-03**: If this milestone adds or renames **primary admin navigation** entries, **labels and ordering** stay aligned with the operator mental model (**billing nouns**, not internal module names), and maintainer-facing **route matrix / README Admin routes** inventory is updated where it would otherwise drift. *(Validated Phase 49 — plan 02, 2026-04-22.)*
- [x] **ADM-04**: All **new or changed user-visible admin strings** introduced in **v1.12** go through **`AccrueAdmin.Copy`** (or an existing copy SSOT module) so **ExUnit** and **Playwright** do not depend on duplicate divergent literals. *(Validated Phase 50 — plans 01–02, 2026-04-22.)*
- [x] **ADM-05**: New or expanded **summary / KPI / card** rows touched in **v1.12** use **`ax-*` layout primitives and theme tokens**; any **intentional** color exceptions are documented in phase notes per **v1.6 UX-04** discipline. *(Validated Phase 50 — plan 01, 2026-04-22.)*
- [x] **ADM-06**: **VERIFY-01** merge-blocking vs **advisory** Stripe lane semantics remain unchanged; every **mounted-admin path materially touched** by **v1.12** gains or extends **Playwright** coverage and **axe** expectations consistent with **v1.6 / v1.7** precedent (extend existing specs where possible). *(Validated Phase 50 — plans 01 + 03, 2026-04-22.)*

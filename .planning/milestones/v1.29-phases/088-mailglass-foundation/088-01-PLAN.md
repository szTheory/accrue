<dep_scope_rationale>
**Why `only: [:dev, :test]` and not `only: [:dev]`:**

`accrue_admin/test/support/live_case.ex:8` calls the `accrue_admin/2` macro with
`allow_live_reload: true`. After Plan 02 lands, that macro expansion contains
`if dev_routes? do ... import MailglassAdmin.Router ... end`. Even though the
`if` is statically truthy in test (so the false branch is elided), the
`import` directive resolves at compile time and requires `MailglassAdmin.Router`
to be loadable in `MIX_ENV=test`.

A `:dev`-only scope would make `MIX_ENV=test mix compile` fail with
"module MailglassAdmin.Router is not loaded". Adding `:test` to the scope keeps
the test-fixture macro expansion resolvable while still excluding mailglass_admin
from `MIX_ENV=prod` (which is what MG-01 actually requires — no admin code in
production releases).
</dep_scope_rationale>

resolve correctly from inside `accrue/mix.exs` or `accrue_admin/mix.exs`. Verified
filesystem layout (Bash `ls -la ~/projects/`):

```
~/projects/

    accrue_admin/   <- accrue_admin/mix.exs lives here
  mailglass/
    mix.exs                        <- :mailglass v0.1.0
    mailglass_admin/
      mix.exs                      <- :mailglass_admin v0.1.0
```

From `accrue/mix.exs`, `:mailglass` is reached at `../../mailglass`.
From `accrue_admin/mix.exs`, `:mailglass_admin` is reached at `../../mailglass/mailglass_admin`.
Both go up TWO levels (out of `accrue/` package, out of `accrue/` repo root) and
then into the sibling `mailglass/` repo.
</path_resolution_note>

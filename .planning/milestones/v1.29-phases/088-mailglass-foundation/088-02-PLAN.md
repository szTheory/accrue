Mailglass docs explicitly recommend the dev-routes wrapping pattern (see
~/projects/mailglass/guides/getting-started.md §3):
```elixir
if Application.compile_env(:my_app, :dev_routes) do
  scope "/dev" do
    pipe_through :browser
    mailglass_admin_routes "/mail"
  end
end
```

the dev-routes guard is reused, no new pipeline or scope is introduced.

<integration_note>
The Mailglass `mailglass_admin_routes/2` macro emits its OWN `live_session` block
internally (it manages session keys, on_mount hooks, and a dedicated LiveView
layout). It is therefore NOT safe to call it inside the existing
`live_session :accrue_admin, ... do` block (router.ex line 52) — Phoenix raises
on nested `live_session` declarations.

Two options exist; this plan picks (B) for minimum diff:

  (A) Lift the `mailglass_admin_routes` call OUT of the live_session and into a
      sibling `scope` block. Cleaner conceptually but increases the diff and
      changes how the macro composes with our existing pipeline.

  (B) Keep the call inside the `if dev_routes? do ... end` block but BREAK OUT
      of the live_session for just that line, by ending the live_session early
      and opening a sibling scope for the Mailglass mount. This is what the
      Mailglass getting-started guide implicitly assumes (the example shows a
      `scope "/dev" do ... end` block at the router level, not nested in a
      live_session).

The plan executes (B) by inserting the Mailglass mount as a sibling `scope`
inside the same `accrue_admin/2` quote block, AFTER the existing
`scope mount_path, as: :accrue_admin do ... end` block closes. The dev-routes
guard wrapping it ensures it only renders in dev (matching `allow_live_reload`).
</integration_note>

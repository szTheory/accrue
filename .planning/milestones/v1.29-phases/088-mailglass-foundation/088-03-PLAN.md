<interfaces>
Mailglass exposes a single migration entry point. Adopters add a thin
wrapper migration file via `mix mailglass.install` (or hand-written),
and Mailglass's Migration helper handles version-tracking dispatch
internally — adopters never write Mailglass DDL by hand.

From `~/projects/mailglass/lib/mailglass/migration.ex`:

  defmodule Mailglass.Migration do
    @doc "Runs all pending migrations up to the latest version."
    @spec up(keyword()) :: :ok
    def up(opts \\ [])

    @doc "Rolls back migrations down to the target version (default: 0)."
    @spec down(keyword()) :: :ok
    def down(opts \\ [])

    @doc "Returns the currently-applied migration version (0 if none)."
    @spec migrated_version(keyword()) :: non_neg_integer()
    def migrated_version(opts \\ [])
  end

Standard adopter wrapper migration shape (per Mailglass docs):

  defmodule MyApp.Repo.Migrations.AddMailglass do
    use Ecto.Migration
    def up, do: Mailglass.Migration.up()
    def down, do: Mailglass.Migration.down()
  end

The `mix mailglass.install` task (Mix.Tasks.Mailglass.Install in
`~/projects/mailglass/lib/mix/tasks/mailglass.install.ex`) generates this
wrapper migration file alongside other host-app wiring. Mailglass's own
test-repo migrations (`~/projects/mailglass/priv/repo/migrations/`) confirm
the three tables that result:

  - mailglass_deliveries     (00000000000001_mailglass_init.exs)
  - mailglass_events         (00000000000001_mailglass_init.exs)
  - mailglass_suppressions   (00000000000001_mailglass_init.exs)

Per-version DDL lives under `Mailglass.Migrations.Postgres.VNN`; Postgres
14+ is required (uses immutable triggers). Accrue's Postgres floor is
14+ (CLAUDE.md), so this is satisfied.
</interfaces>

Phase 88's narrowing of MG-03 to "documentation + verification, not installer
auto-generation" is sourced from the orchestrator's plan-phase invocation
context, which is itself traceable to the milestone research document.

**Source 1 — Orchestrator plan-phase context** (provided to this planner at

+ verification, not necessarily migration auto-generation."

**Source 2 — Milestone research §3E** (`.planning/research/v1.29-mailglass-integration.md`):
the research document is permissive on the install path, listing as acceptable
either "execute `mix mailglass.install` automatically" OR "instruct host
applications to run the Mailglass migrations." The phase orchestration chose
the latter (instruct host applications) because:

  - `mix accrue.install` orchestration changes are bigger than Phase 88's scope
    (touch the installer's task module, generator templates, and existing
    install-time checkpoints — none of which are in `files_modified` for any
    Phase 88 plan).

    REQUIREMENTS.md MG-03 ("Update Accrue's installation instructions and/or
    `mix accrue.install`...") because the wording is explicitly permissive
    (the "and/or" allows either branch alone).
  - Phase 88's success criterion #1 ("Developer can run Mailglass migrations
    in the host application successfully") is provable by docs + verification
    transcript without `mix accrue.install` orchestration.

**Deferral plan**: Auto-generation through `mix accrue.install` is deferred to
either (a) a follow-on quick task between Phase 88 and Phase 89, or (b) Phase 89
or 90 if adopters explicitly request it during dogfooding. Either path is
non-blocking for Phase 88 acceptance.
</scope_decision_source>

# Requirements: Accrue v1.5 Adoption proof hardening

**Defined:** 2026-04-18  
**Core value:** A Phoenix developer can install Accrue + its companion admin UI, and launch a real SaaS with subscription billing on day one — complete, production-grade, with idiomatic Elixir DX, strong domain modeling, tamper-evident audit ledger, great observability, and zero breaking-change pain through at least the first major version.

**Milestone priority (phased A→B):** (A) Make Stripe **test-mode** provider parity and its relationship to VERIFY-01 obvious to evaluators and maintainers. (B) Document the **B2C-shaped API** vs **org-first LiveView** split in the canonical host so adoption questions have a single matrix. (C) Ship an **evaluator screen-recording script** (checklist, no committed video binaries).

## v1.5 requirements

### Adoption proof documentation

- [x] **PROOF-01**: Cross-link Fake-first VERIFY-01 / `mix verify.full` with the advisory Stripe test-mode lane (`mix test.live`, workflow job `live-stripe`); clarify naming in CI and `guides/testing-live-stripe.md` where contributors look first.
- [x] **PROOF-02**: Add a durable **adoption proof matrix** for `examples/accrue_host` that states which archetypes are covered by bounded ExUnit, full tests, and Playwright — including explicit note that `/app/billing` LiveView is org-scoped while user billable flows are proven via `Billing.subscribe(user, …)` in `billing_facade_test.exs`.
- [x] **PROOF-03**: Add an **evaluator walkthrough script** (markdown checklist) for screen recordings, linked from the host README alongside the existing screenshot / artifact story.

Create `Accrue.Billable` macro module per D2-04. When a host schema calls `use Accrue.Billable`, inject:

1. **`has_one :accrue_customer`** association:
   ```elixir
   has_one :accrue_customer, Accrue.Billing.Customer,
     foreign_key: :owner_id,
     references: :id,  # host schema PK -- converted to string at query time
     where: [owner_type: ^billable_type]
   ```
   where `billable_type` is computed at macro-expansion time from `opts[:billable_type]` or `__MODULE__ |> Module.split() |> List.last()` per D2-02.

   IMPORTANT: The `has_one` must use the string representation of the owner's PK as the foreign_key match. Since Customer.owner_id is a string column (D2-01), the association needs a custom `references:` pointing to the host PK. Because host PKs can be integer, UUID, or ULID, the association uses `references: :id` and the actual query/preload joins on `to_string(host.id) == customer.owner_id`. Use `Ecto.Association.has_one`'s `:where` option to scope by owner_type.

   NOTE: The exact `references:` field may need adaptation if the host uses integer PKs. The safest approach is to define `references: :id` and handle type coercion in the Billing context's fetch/create path rather than in the association definition. Document this in the moduledoc.

2. **`__accrue__(:billable_type)`** reflection callback returning the computed billable_type string.

3. **`customer/1`** convenience function that delegates to `Accrue.Billing.customer(struct)` per D2-06 (lazy fetch-or-create).

Accept `use Accrue.Billable, billable_type: "User"` as override for rename-safety per D2-02. The `@moduledoc` must include the rename-safety warning from D2-02.

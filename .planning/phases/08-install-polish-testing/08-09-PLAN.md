<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| test auth adapter to admin authorization | `Accrue.Auth.Mock` can satisfy admin checks in tests and must not become production auth. |
| process-local mock state to async tests | One test process can set a current user without leaking that user to another test process. |
| audit callback to test assertions | Test audit events are sent to the calling process, not persisted as production audit records. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-09-01 | Elevation of Privilege | `Accrue.Auth.Mock.require_admin_plug/0` | mitigate | Gate pass-through on `Accrue.Auth.admin?/1`; non-admin or missing users raise `Accrue.ConfigError` in tests. |
| T-08-09-02 | Elevation of Privilege | `Accrue.Auth.Mock` in production | mitigate | `ensure_test_env!/0` raises `Accrue.ConfigError` when `:accrue, :env` is `:prod`; tests assert the exact production refusal message. |
| T-08-09-03 | Information Disclosure | mock current user state | mitigate | Store mock users in the current process dictionary only; do not use ETS, persistent_term, application env, or global state for test users. |
| T-08-09-04 | Repudiation | `log_audit/2` | accept | In mock mode audit is a test assertion signal via `{:accrue_auth_audit, user, event}` to `self()`; production audit remains owned by real auth adapters. |

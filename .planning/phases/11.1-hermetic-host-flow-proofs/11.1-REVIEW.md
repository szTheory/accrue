# Phase 11.1: Code Review Report

**Reviewed:** 2026-04-16T20:48:25Z
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Reviewed the host proof tests, their shared case template, and the phase UAT shell script. The main problems are test-safety and behavioral coverage: the shared cleanup helper deletes far more data than the proof flows own, and the new replay/idempotency tests stop short of asserting the queueing behavior that makes those flows actually work.

## Warnings

### WR-01: Shared proof cleanup removes non-proof billing rows

**File:** `examples/accrue_host/test/support/host_flow_proof_case.ex:91-145`
**Issue:** The helper claims to delete only rerun-collision rows, but the queries are much broader than that. `delete_proof_jobs!/0` removes every `Accrue.Webhook.DispatchWorker` job, `delete_proof_events!/0` removes every event with the listed types, and `delete_proof_subscription_items!/0` removes every `price_basic` item. That makes these proofs depend on global table state and can hide regressions or invalidate unrelated host fixtures that happen to use the same worker, event types, or price ID.
**Fix:**
```elixir
# Scope cleanup to records created by the proof flows instead of global types.
proof_customer_ids =
  from(customer in Customer,
    where:
      customer.email in ^@cleanup_customer_emails or
        like(customer.email, "host-%@example.test"),
    select: customer.id
  )

proof_subscription_ids =
  from(subscription in Subscription,
    where: subscription.customer_id in subquery(proof_customer_ids),
    select: subscription.id
  )

Repo.delete_all(
  from(job in Oban.Job,
    where:
      job.worker == "Accrue.Webhook.DispatchWorker" and
        fragment("?->>'webhook_event_id' LIKE 'evt_host_%'", job.args)
  )
)

Repo.delete_all(
  from(item in SubscriptionItem,
    where: item.subscription_id in subquery(proof_subscription_ids)
  )
)
```

### WR-02: Replay test never proves the webhook is re-enqueued

**File:** `examples/accrue_host/test/accrue_host_web/admin_webhook_replay_test.exs:116-120`
**Issue:** After clicking replay, the test only checks that the row goes back to `:received` and that an audit event exists. A regression where replay updates status and audit logging but forgets to enqueue a fresh `DispatchWorker` job would still pass, leaving the replayed webhook stuck.
**Fix:**
```elixir
replay_html = render_click(element(replay_view, "[data-role='replay-single']"))
assert replay_html =~ "Webhook replay requested."

assert Repo.exists?(
         from(job in Oban.Job,
           where:
             job.worker == "Accrue.Webhook.DispatchWorker" and
               fragment("?->>'webhook_event_id' = ?", job.args, ^to_string(webhook.id))
         )
       )
```

### WR-03: Idempotency coverage stops before the processed-event case

**File:** `examples/accrue_host/test/accrue_host_web/webhook_ingest_test.exs:45-83`
**Issue:** The duplicate delivery assertion runs only while the webhook is still `:received`. That misses the higher-risk case where Stripe retries after the first event has already been processed successfully. An implementation that wrongly re-enqueues or re-handles duplicates after `:succeeded` would still pass this test.
**Fix:**
```elixir
assert :ok = DispatchWorker.perform(%Oban.Job{args: %{"webhook_event_id" => webhook.id}, attempt: 1, max_attempts: 25})

duplicate_after_success = post_webhook(payload, signature)
assert duplicate_after_success.status == 200
assert Repo.aggregate(WebhookEvent, :count) == 1
assert Repo.aggregate(Oban.Job, :count) == 1

assert Repo.aggregate(
         from(event in Event,
           where:
             event.type == "host.webhook.handled" and
               event.caused_by_webhook_event_id == ^webhook.id
         ),
         :count
       ) == 1
```

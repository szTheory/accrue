# Phase 11.1: Code Review Report

**Reviewed:** 2026-04-16T20:52:39Z
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Reviewed the host-flow proof tests, shared test helpers, and the host UAT shell gate after the regression-gate fix. The direct test coverage additions are directionally good, and I did not find a production security issue in this scope. The remaining concerns are both test-reliability problems in the shared setup helpers: they rely on globally mutable state without enforcing serial execution, and one helper performs per-test DDL against the append-only events table.

## Warnings

### WR-01: Shared case templates remain unsafe for async callers

**File:** `examples/accrue_host/test/support/accrue_case.ex:29-37`, `examples/accrue_host/test/support/host_flow_proof_case.ex:54-62`
**Issue:** Both case templates accept `tags[:async]` through `setup_sandbox/1`, but they also unconditionally call `Accrue.Processor.Fake.start_link/1` and `Accrue.Processor.Fake.reset/0` on a fixed-name singleton. `Accrue.Processor.Fake` is globally registered, so any future test that uses either helper with `async: true` will race on reset and cross-contaminate processor state. The reviewed tests all use `async: false`, but the helper itself does not enforce that contract, so this is a latent flake/regression source.
**Fix:**
```elixir
setup tags do
  if tags[:async] do
    raise "AccrueHost.AccrueCase requires async: false because Accrue.Processor.Fake is global"
  end

  AccrueHost.DataCase.setup_sandbox(tags)
  ...
end
```

Apply the same guard in `HostFlowProofCase`, or refactor the fake processor to support per-test named instances instead of a singleton.

### WR-02: Per-test trigger disabling can block or perturb concurrent DB tests

**File:** `examples/accrue_host/test/support/host_flow_proof_case.ex:99-110`
**Issue:** `delete_proof_events!/0` runs `ALTER TABLE accrue_events DISABLE TRIGGER ...` and reenables it in every test setup. In PostgreSQL this is transactional DDL that acquires strong table locks. Because ExUnit may run unrelated async tests at the same time, this helper can block other tests that insert into `accrue_events` or temporarily change table behavior outside the current test's intent. That makes the proof suite a source of cross-test nondeterminism.
**Fix:**
```elixir
# Prefer isolating proof rows by unique ids and deleting only those rows
# without changing table-level trigger state inside test setup.

@cleanup_event_actor_ids ["evt_host_", "evt_host_replay_"]

Repo.delete_all(
  from(event in Event,
    where:
      event.type in ^@cleanup_event_types and
      like(event.actor_id, "evt_host_%")
  )
)
```

If append-only semantics prevent ordinary deletes, move this cleanup to an isolated DB reset step outside concurrent ExUnit execution, or run these proof tests in a dedicated suite that owns the database exclusively.

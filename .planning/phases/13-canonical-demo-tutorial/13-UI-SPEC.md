# Phase 13 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | Phoenix LiveView function components plus existing `accrue_admin` package components |
| Icon library | none new in Phase 13 |
| Font | system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", sans-serif |

Phase 13 inherits the existing `examples/accrue_host` and `accrue_admin` UI. Do not introduce a new visual system, new component library, or a redesign pass. This phase only locks tutorial labels, command ownership, stable smoke-visible copy, and the host/admin interaction path already proved by tests.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline badge padding, compact metadata spacing |
| sm | 8px | Label gaps, inline action gaps, eyebrow spacing |
| md | 16px | Default card gaps, body spacing, command block separation |
| lg | 24px | Card padding, section spacing, shell gaps |
| xl | 32px | Page vertical padding, top-level section starts |
| 2xl | 48px | Empty-state padding, major tutorial section breaks |
| 3xl | 64px | Reserved for page-level separation only; do not add beyond current layouts |

Exceptions: Existing host buttons may keep 12px vertical padding. Do not add any new spacing token outside the 4px grid.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FAFBFC` / inherited admin base surface | Host page background and documentation page background tone |
| Secondary (30%) | `#FFFFFF` | Cards, empty states, mounted admin panels, code-surface containers |
| Accent (10%) | `#5D79F6` | Primary billing CTA, tutorial links to actionable paths, active state highlights, focus-accented interactive affordances |
| Destructive | `#111418` | Existing cancel confirmation action only; do not add new destructive color work in Phase 13 |

Accent reserved for: `Go to billing`, `Start subscription`, active plan badge, tutorial mode labels when visually emphasized, admin active navigation/tab highlights, and focus treatments. Do not use accent as generic body-text decoration or for broad documentation callout backgrounds.

## Phase 13 Surface Scope

In scope:

- `examples/accrue_host/README.md` as the canonical clone-to-running path.
- `accrue/guides/first_hour.md` as the package-facing mirror of the same ordered story.
- `examples/accrue_host` subscription page and mounted `/billing` admin path only where copy or smoke-visible expectations must stay stable.
- Verification-command naming and how those commands are explained to the user.

Out of scope:

- Root README adoption-front-door redesign.
- New admin IA, new billing flows, new dashboards, or a new color/token system.
- Hosted demo, live Stripe teaching flow, or broad docs information architecture work.

## Tutorial And Command Contract

| Surface | Required Contract |
|---------|-------------------|
| README and First Hour information order | Show the Phoenix-order path first: prerequisites, `cd examples/accrue_host`, `mix setup`, `mix phx.server`, create the first subscription, post one signed webhook proof, inspect `/billing`, then run focused verification. |
| `First run` | This is the canonical path. It must use public host boundaries only: generated `AccrueHost.Billing`, the real `/webhooks/stripe` endpoint, and the mounted admin UI. |
| `Seeded history` | This is a secondary evaluation path for deterministic replay/history/browser coverage. It must be visibly labeled as evaluation setup, not the public teaching path. |
| `mix verify` | Describe as the focused local proof suite for the tutorial story. It is maintainer- and evaluator-friendly, but it appears after the user understands the first-run flow. |
| `mix verify.full` | Describe as the CI-equivalent full local gate. It can include compile, assets, dev boot, and browser smoke, but it must not replace the first-run path in the tutorial narrative. |
| `bash scripts/ci/accrue_host_uat.sh` | Keep as a thin repo-root wrapper around the same full contract. It is maintainer-facing and may be referenced after `mix verify.full`, not before. |
| Hex smoke | Keep explicitly separate from the canonical demo path. Never mix Hex-smoke instructions into `First run`. |

Documentation structure contract:

- Use `First run` and `Seeded history` as peer section labels.
- Use fenced `bash` blocks for commands and fenced `elixir` blocks for code.
- Keep prose human-written. Manifest-backed tests may enforce labels, ordering, and parity, but the docs themselves must remain authored prose.

## Stable Browser And Admin Smoke Surfaces

These strings and interaction points are test-visible and must remain stable unless the phase intentionally updates both docs and smoke coverage together.

| Surface | Stable Expectation |
|---------|--------------------|
| Home navigation | `Go to billing` link remains the host-side entry into the billing demo flow. |
| Subscription page display | `Choose a plan` remains the page heading for the host billing screen. |
| First-run empty state | `No billing activity yet` remains the first billing-state heading before a subscription exists. |
| Subscription success state | `Current subscription` remains the summary heading after a plan starts. |
| Admin dashboard | `Local billing projections at a glance` remains the stable smoke-visible heading proving `/billing` mounted successfully. |
| Admin events index | `Append-only billing and admin activity` remains the heading for the audit/history view. |
| Webhook replay feedback | `Webhook replay requested.` remains the replay confirmation flash. |
| Retry metadata | Attempt copy keeps the `Attempt N/25` format for dead/discarded webhook state inspection. |
| Admin access control | Anonymous users and signed-in non-admin users are redirected away from `/billing`; only billing admins can mount it. |

## Interaction Contract

| Area | Contract |
|------|----------|
| Tutorial progression | The user should move from host app setup to first success before seeing the full gate. Do not front-load CI wrapper commands. |
| Public vs private boundaries | `First run` teaches only supported host-owned surfaces. Private seeding is allowed only under `Seeded history` and must be labeled as such. |
| Subscription proof | The canonical story is start one Fake-backed subscription, post one signed webhook event, inspect the result in `/billing`, then run focused proofs. |
| Cancellation flow | Keep cancellation as a secondary proof only. It may stay in focused tests/browser smoke, but it must not displace the first subscription and admin inspection story in docs. |
| Admin inspection | The inspection path must preserve customer/subscription detail, webhook history, replay action, and resulting audit history as the stable trust proof. |

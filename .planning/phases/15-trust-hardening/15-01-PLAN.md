<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| webhook request -> raw-body verification | Untrusted Stripe-style input crosses into Accrue's verification, persistence, and queueing path. |
| host auth/session -> mounted admin routes | Host-owned session state controls access to `/billing` and replay actions. |
| admin operator -> replay + audit | A privileged action can requeue webhook work and must stay auditable. |
| generated host code -> package-owned library code | Installer output becomes host-owned after generation and can drift from package assumptions. |
| docs/errors/logs/artifacts -> public readers | Public docs, diagnostics, and retained artifacts can leak secrets or PII if the wording or retention policy regresses. |
| public issue intake -> maintainers | Public support surfaces must not invite production payloads, tokens, secrets, or customer data. |

Policy: High-severity threats are release-blocking for Phase 15 and cannot be accepted. Only low- or medium-severity items may be accepted, and each accepted item must carry a concrete evidence-based rationale.

| T-15-01 | T | `.planning/phases/15-trust-hardening/15-TRUST-REVIEW.md` | High | V1.1, V1.14 | mitigate | Link every boundary to exact repo evidence and explicitly label host-owned and advisory areas so webhook/auth/admin/replay/generated-host assumptions cannot be implied or hidden. |

</threat_model>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| repo CI workflow -> release/support contract | Workflow labels and matrix semantics are the operational source of truth for what is required versus advisory. |
| host integration job -> retained artifacts | CI decides which browser reports, traces, screenshots, and logs are kept and exposed to maintainers. |
| advisory provider-backed lane -> maintainers | An advisory cell can be misread as supported-blocking if `continue-on-error` or job naming drifts. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-11 | R | `.github/workflows/ci.yml` compatibility matrix | mitigate | Keep the support floor, primary target, and forward-compat cells in one workflow and encode advisory behavior with matrix `continue-on-error`, not tribal knowledge. |
| T-15-12 | D | host integration lane | mitigate | Continue running the canonical host UAT wrapper and keep it in the required deterministic path so trust-smoke and browser gates cannot drift into a side workflow. |
| T-15-13 | I | retained artifacts | mitigate | Keep Playwright reports, traces, and logs failure-only, and limit success retention to the compact canonical screenshot set produced by the browser trust flow. |
| T-15-14 | S | advisory live Stripe lane | mitigate | Preserve `live-stripe` as explicitly advisory with `continue-on-error: true` and workflow comments that route readers back to the public provider-parity guide. |
</threat_model>

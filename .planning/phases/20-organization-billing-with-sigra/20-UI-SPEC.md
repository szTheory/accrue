## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gaps, eyebrow-to-heading spacing |
| sm | 8px | Label-to-field spacing, compact row gaps |
| md | 16px | Default control spacing, flash padding, stacked actions |
| lg | 24px | Card padding, section padding |
| xl | 32px | Page section gaps, KPI-to-content separation |
| 2xl | 48px | Major page breaks |
| 3xl | 64px | Top-level page breathing room only |

Exceptions: minimum 44px hit area for all clickable icon-only or compact action controls; no other exceptions.

Implementation rules:
- Host org billing content keeps the existing single-column mobile stack and may expand to two columns only for plan/action sections at `md+`.
- Admin detail pages keep current `ax-page`, `ax-card`, `ax-kpi-grid`, and `ax-grid-2` layout structure; do not nest a new card inside an existing confirmation or warning card.
- Confirmation panels, denial banners, and owner badges must use existing page spacing rhythm and may not introduce ad hoc gaps.

Implementation rules:
- Reuse existing host/admin theme tokens. Do not introduce a new palette.
- No new decorative gradients, orbs, glows, or marketing-style background treatments for Phase 20 additions.
- Cross-org denial and ambiguous ownership states use warning/error semantics, not accent.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Start organization subscription |
| Empty state heading | No organization billing activity yet |
| Empty state body | Billing records appear after an organization starts a subscription or a webhook is processed. Start the organization subscription or review webhook activity for this organization. |
| Error state | We couldn't complete that billing action for the active organization. Check organization access, billing setup, or webhook processing, then try again. |
| Destructive confirmation | Cancel organization subscription: Confirm cancellation before ending organization access. |

Additional locked copy:

| Element | Copy |
|---------|------|
| Active organization label | Active organization |
| Active organization helper | Billing actions apply to the active organization only. |
| Member-without-admin permission message | Billing is managed by organization admins. You can review the current billing state, but you can't change it. |
| No active organization message | Select an active organization before managing billing. |
| Cross-org denial | You don't have access to billing for this organization. |
| Ambiguous webhook ownership | Ownership couldn't be verified for this webhook. Replay is unavailable until the linked billing owner is resolved. |
| Single replay confirmation | Replay webhook for the active organization? |
| Bulk replay confirmation | Replay {count} failed or dead webhook rows for the active organization? |
| Replay success | Replay requested for the active organization. |
| Replay blocked | Replay is blocked because this webhook isn't linked to a billable row in the active organization. |

Copy rules:
- Copy must name the billing state or action, not the interface.
- Use “organization” consistently in host copy and “owner” only where the data model is being exposed in admin detail.
- Do not expose raw authorization jargon such as “tenant,” “scope,” or “policy” in user-facing text.

## Host Billing Contract

### Ownership and scope

- The canonical host billing surface must derive the billable organization from server-side active organization context on `current_scope`. It must never trust a client-selected organization ID from params, hidden inputs, or event payloads.
- Display the active organization as read-only context above billing actions using the locked label `Active organization`.
- If no active organization is present, disable mutation actions and render the locked `No active organization message`.
- If the viewer is an organization member but not an organization admin, render current billing state read-only and show the locked `Member-without-admin permission message`.

### Layout

- Reuse the existing host billing structure: page header, current state block or empty state block, tax/location repair block, and plan/action list.
- The active organization context plus current billing state block is the first-read focal area on the page.
- Do not create a separate marketing page or a detached org-selector page for this phase.
- Keep one visual container per section. No card-inside-card compositions.

### States

- Empty org billing state uses the locked empty-state heading/body above the plan list.
- Existing subscription state shows organization-owned customer and subscription summary with the same information density as the current user billing proof.
- Tax-location repair stays in the same host flow and remains subordinate to the active organization context.
- Cancel flow stays two-step: request cancel -> explicit confirmation copy -> confirm or keep subscription.

### Interaction rules

- All org-scoped submit actions must use the host facade layer, not direct schema queries.
- Buttons that mutate billing must disable while the LiveView event is in flight.
- Success and error feedback uses the existing flash mechanism; keep one clear billing outcome per flash.
- Do not allow users to switch organizations inside the billing form itself unless that control already comes from upstream Sigra shell context outside this phase.

## Admin Owner-Scoped Contract

### Common behavior

- Customer, subscription, and webhook detail routes must prove row ownership server-side before rendering any row content.
- Unauthorized direct links must redirect to the relevant scoped index page and show the locked `Cross-org denial` flash copy.
- Once denied, do not render partial row metadata, IDs, payload fragments, counts, or replay controls.

### Customer detail

- Keep the current detail layout and KPI structure.
- The `Owner` KPI remains required. For organization-owned rows it is the primary ownership cue and must stay above related tabs/content.
- If the customer owner is an organization, any linked subscription/invoice/charge/payment-method lists inherit that same organization scope automatically; no org switcher appears on the page.

### Subscription detail

- Keep the existing summary KPIs, timeline, and staged confirmation pattern.
- Add owner context through the linked customer rather than a new standalone selector.
- Subscription actions remain staged, then confirmed. Confirmation copy must name the action and keep owner scope implicit through the page context.
- Cross-org denial happens before any staged action is created.

### Webhook detail and bulk replay

- Webhook detail and bulk replay actions must prove organization ownership through linked local billing rows or event causality before showing actionable replay controls.
- If ownership is ambiguous, show the locked `Ambiguous webhook ownership` copy in warning treatment and disable replay.
- Single replay and bulk replay confirmations must include active-organization wording from the locked copy table.
- Bulk replay counts must be scoped counts only. Never confirm against a global DLQ total when the operator is org-scoped.

### Ambiguous ownership behavior

- When a webhook row cannot be linked confidently to a single owner in the active organization, detail access may still render non-sensitive operational context only if Phase 20 implementation can prove that context is org-safe; otherwise redirect with the denial flash.
- Replay is always blocked on ambiguous ownership.
- Do not ask the operator to choose an organization manually to resolve ambiguity.

## Accessibility and Responsive Contract

- All flashes and replay/cancel confirmation outcomes must be announced through existing LiveView-accessible flash patterns; keep messages concise enough for screen-reader repetition.
- Focus-visible styles must use existing theme tokens. Do not remove outlines.
- Buttons, tabs, filters, and row links must remain keyboard reachable in DOM order.
- Confirmation panels must mount adjacent to the triggering action block so focus does not jump across the page unexpectedly.
- Long IDs, org names, and webhook types must wrap rather than overflow or expand the layout horizontally.
- KPI rows collapse cleanly on small screens; no clipped labels, counts, or badges.
- Table/list views that already support card rendering on narrow screens must keep the same responsive fallback.
- During LiveView loading, disable the triggering action and preserve the surrounding layout height so buttons and adjacent text do not jump.
- Respect `prefers-reduced-motion`; no new animated transitions beyond existing theme timing.

---

## Visual Constraints

- Stay inside the existing host and admin themes. This phase is not allowed to introduce a new visual identity.
- No landing page, no hero, no decorative illustrations, and no ornamental background effects.
- Border radius for any new or modified Phase 20 cards, buttons, badges, inputs, and confirmation panels must be 8px or less.
- Avoid pill-shaped new controls. If a shared legacy class is reused unchanged elsewhere, do not spread that shape into new Phase 20-specific elements.
- Keep text content product-facing only.

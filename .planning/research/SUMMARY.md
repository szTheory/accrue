# Project Research Summary — Accrue

**Project:** Accrue
**Domain:** Open-source Elixir/Phoenix payments & billing library (Stripe-first, with companion LiveView admin UI)
**Researched:** 2026-04-11
**Confidence:** HIGH

## Executive Summary

Accrue is a batteries-included Elixir/Phoenix billing library filling the ecosystem gap that Pay fills for Rails, Cashier fills for Laravel, and dj-stripe fills for Django. Research across stack, features, architecture, and pitfalls converges on a single build thesis: **ship a headless billing kernel (`accrue`) plus a companion LiveView admin UI (`accrue_admin`) as sibling Hex packages in one monorepo, Stripe-only, PostgreSQL-only, complete on first release**. Every major design decision locked in `PROJECT.md` is validated by the research, with one exception that is the single most important sequencing constraint: **`lattice_stripe` (Accrue's Stripe HTTP dependency, v0.2.0) has no Billing coverage yet** — no `Subscription`, `Invoice`, `Price`, `Product`, or `Meter` resources. Accrue cannot ship subscriptions until those land upstream or are written as fallback `%LatticeStripe.Request{}` calls inside Accrue itself.

The recommended approach is strictly topological. Build foundations first — config, `Accrue.Error` hierarchy, telemetry conventions, the `Accrue.Money` value type, the `Accrue.Processor` behaviour, and critically the Fake Processor, which is the **primary** test surface for every downstream feature, not a test-layer afterthought. Then schemas and the webhook pipeline, then the core subscription lifecycle, then advanced billing + webhook hardening, then Connect, then email/PDF rendering, then the admin UI, then the installer + testing polish, then release. The event ledger (`accrue_events`) is DB-trigger-enforced append-only, written transactionally alongside state mutations, and bridges to `Sigra.Audit` when present. Users interact through a thin `MyApp.Billing` context facade generated by `mix accrue.install`, never directly against `lattice_stripe`.

Key risks concentrate in four areas: (1) the lattice_stripe Billing gap, which blocks everything past Phase 2 until resolved; (2) Stripe semantic footguns (`cancel_at_period_end` vs `status`, `incomplete` 23-hour expiry, zero/three-decimal currencies, proration defaults, idempotency key reuse, refund fee asymmetry, webhook ordering/duplication) — all catalogued with explicit per-feature mitigations; (3) the processor-abstraction trap that sank Cashier v1 ("beautiful lie" of false multi-processor parity) — mitigated by having the Fake Processor be the behaviour's second real implementation before any non-Stripe work; and (4) optional-dep compile hygiene under `--warnings-as-errors`, requiring the `no_warn_undefined` + `Code.ensure_loaded?/1` + runtime adapter lookup pattern that `sigra` already uses.

## Key Findings

### Recommended Stack

Versions verified against Hex.pm API on 2026-04-11. Core runs on Elixir 1.17+, OTP 27+, Phoenix 1.8+, Ecto 3.13+, PostgreSQL 14+. Full `mix.exs` template and compatibility matrix in `STACK.md`.

**Required runtime deps:**
- `:lattice_stripe ~> 0.2` — Stripe HTTP wrapper (sibling lib). **CRITICAL GAP:** no Subscription/Invoice/Price/Product/Meter coverage yet.
- `:oban ~> 2.21` — webhook async + scheduled emails; community edition is sufficient, no PG extensions required.
- `:ecto ~> 3.13` + `:ecto_sql ~> 3.13` + `:postgrex ~> 0.22` — domain modeling.
- `:swoosh ~> 1.25` + `:phoenix_swoosh ~> 1.2` — email behind `Accrue.Mailer` facade.
- `:mjml_eex ~> 0.13` — responsive email templates. **NOT `swoosh_mjml`** — that package does not exist on Hex.
- `:chromic_pdf ~> 1.17` — default PDF adapter; host owns the ChromicPDF supervision, Accrue only calls into it.
- `:nimble_options ~> 1.1` — config schema + validation + ExDoc auto-gen.
- `:decimal ~> 2.0` — pin explicitly (already transitive) because money math correctness depends on it.
- `:jason ~> 1.4`, `:telemetry ~> 1.3`.

**Optional deps** (all conditional-compiled via `no_warn_undefined` + `Code.ensure_loaded?/1`): `:sigra ~> 0.1`, `:opentelemetry ~> 1.7`, `:opentelemetry_api ~> 1.4`, `:telemetry_metrics ~> 1.1`.

**Test library: Mox, decisively.** Accrue ships three behaviours (`Accrue.Auth`, `Accrue.Mailer`, `Accrue.PDF`) whose entire purpose is to be mocked — canonical Mox fit per Valim's "Mocks and Explicit Contracts." Both sibling libs use Mox; style-matching reduces cognitive overhead.

**What NOT to use:** `stripity_stripe` (2019 API pin), `bamboo` (maintenance-mode), `poison` (abandoned), `exq`/`verk`/`rihanna` (Oban is the standard), `commanded`/`eventstore` (rejected in PROJECT.md), `pdf_generator` (wraps archived wkhtmltopdf with CVEs), `pg_uuidv7` extension (`gen_random_uuid()` suffices).

### Architecture Approach

Headless billing kernel + companion LiveView admin UI, one monorepo, two independently versioned Hex packages. Strict one-directional layering: host web → generated `MyApp.Billing` facade → `Accrue.Billing.*` public API → `Accrue` behaviours → adapters → internal engines (webhook, events, schemas) → external deps → PostgreSQL. Core `accrue` never references `Phoenix.LiveView`, `Phoenix.Component`, or `Phoenix.Router`; that's `accrue_admin`'s exclusive territory. `Phoenix.HTML` and `Phoenix.Template` are allowed in core for email/PDF rendering because they work without LiveView.

**Major components:**

1. **`Accrue.Billing.*` context + schemas** — polymorphic Customer, Subscription (state machine), Invoice, Charge, PaymentIntent, SetupIntent, PaymentMethod, Refund, Coupon, PromotionCode, SubscriptionSchedule, Price, Product, plus `Accrue.Money` zero/three-decimal-safe value type. Pure Ecto schemas; Stripe orchestration lives in context functions using `Ecto.Multi` for atomic writes with event-ledger entries.
2. **`Accrue.Processor` behaviour + adapters** — callbacks for every Stripe operation. Returns raw maps (stored verbatim in `data` jsonb per dj-stripe convergence). First-party adapters: `Accrue.Processor.Stripe` (delegates to lattice_stripe) and `Accrue.Processor.Fake` (in-memory ETS state, deterministic IDs, test clock, scriptable responses). Idempotency keys derived deterministically by Accrue from `{operation, subject_id, canonical_params}`.
3. **Webhook pipeline** — `Accrue.Webhook.Plug` (raw-body capture **scoped** to webhook routes only), Verifier, `accrue_webhook_events` schema, Idempotency (insert-or-skip on `processor_event_id`), Dispatcher (Oban enqueue in single `Repo.transact` with the insert), Worker, user-implemented Handler behaviour, DefaultHandler for built-in state reconciliation, Replay/DLQ tooling.
4. **Append-only event ledger** — `accrue_events` table with DB-trigger-enforced immutability, `Events.Recorder` (Ecto.Multi-friendly insert), Upcaster (jsonb `schema_version` evolution), Query API (timeline, state-as-of, analytics bucketing), SigraBridge (one-way passthrough to `Sigra.Audit` when present).
5. **Email + PDF rendering** — `Accrue.Mailer` behaviour with `Mailer.Swoosh` + `Mailer.Test`, `Accrue.PDF` behaviour with `PDF.ChromicPDF` + `PDF.Test` + `PDF.Null`, HEEx templates shipped in `lib/` and renderable in both email HTML and PDF contexts.
6. **`Accrue.Auth` behaviour** — with `Auth.Default` (dev-only fallback; fails closed in prod) and `Accrue.Integrations.Sigra` conditionally compiled via `Code.ensure_loaded?(Sigra)`.
7. **`accrue_admin` LiveView package** — depends on `accrue`, never reverse. Reads schemas directly via host Repo for listing/pagination but mutates exclusively through `Accrue.Billing.*` context functions so admin actions and webhook handlers converge on one write path.
8. **`mix accrue.install` generator** — writes migrations, `MyApp.Billing` context facade, router mounts, admin routes, webhook endpoint Plug; idempotent re-run; auto-wires sigra if present.

### Critical Pitfalls

Top 10 traps every phase must defuse (full catalog in `PITFALLS.md`):

1. **`cancel_at_period_end` vs `status` confusion** — two orthogonal flags. Fix: three canonical predicates (`active?/1`, `canceling?/1`, `canceled?/1`); never expose raw `status` as an access-control primitive.
2. **`incomplete` subscriptions silently expiring after 23 hours** — first-invoice-fail creates a 23h window. Fix: `active?/1` excludes `incomplete`; subscribe to `invoice.payment_action_required`; emit ops telemetry on the transition.
3. **Zero-decimal and three-decimal currencies** — `amount * 100` catastrophic for JPY/KRW/VND/CLP and BHD/KWD/JOD. Fix: `Accrue.Money` wraps `{minor_amount, currency}`, rejects bare integers at API boundaries.
4. **Proration default behavior** — varies across endpoints and API versions. Fix: `swap_plan/3` takes an **explicit** `:proration` option, never inherits; `preview_upcoming_invoice/2` exists.
5. **Idempotency key replay with changed params** — Stripe holds keys 24h pinned to full body. Fix: deterministic keys as hashes of `{operation, subject_id, canonical_params}`; Oban retries attach the key to job args.
6. **Refund fee asymmetry** — Stripe does NOT return processing fees on refunds. Fix: `Refund` schema includes `stripe_fee_refunded_amount` and `merchant_loss_amount`; admin UI surfaces breakdown.
7. **Webhook ordering / duplicate delivery** — at-least-once unordered. Fix: handler re-fetches current object; resolve state from newest event per subject (by Stripe `created`); layered idempotency.
8. **Raw-body plug scoping** — mounting globally kills streaming on every non-webhook route. Fix: scoped pipeline; installer wires it correctly.
9. **Trace/actor gaps in event ledger** — naive schema leaves audit log useless at the moment it's needed. Fix: `actor_type` NOT NULL enum (`user | system | webhook | oban | admin`), OTel trace_id/span_id captured on every write.
10. **Optional-dep compile warnings** — sigra-absent compilation under `--warnings-as-errors` fails without `no_warn_undefined` + `Code.ensure_loaded?/1`. Fix: follow sigra's own pattern verbatim.

## Implications for Roadmap

Research converges on a strictly topological **9-phase build with a conditional Phase 0 prefix**. Build order is non-negotiable — each layer literally cannot compile without its predecessor.

### Phase 0 (Conditional): lattice_stripe Billing Contribution

**Rationale:** Single most important sequencing constraint. lattice_stripe v0.2 has NO Subscription/Invoice/Price/Product/Meter coverage. Accrue cannot ship past Phase 2 without this. Two paths: **(A)** upstream contribution to lattice_stripe 0.3 (preferred) or **(B)** in-Accrue fallback `%LatticeStripe.Request{}` calls to be upstreamed later.
**Decision point:** Roadmapper picks A or B at project kickoff.
**Delivers:** lattice_stripe 0.3 OR Accrue-internal fallback modules.
**Addresses:** Unblocks PROC-002, BILL-010…103, and everything invoice-related.

### Phase 1: Foundations

**Rationale:** Everything depends on these. `Accrue.Money` and Fake Processor must land first — Money because any schema built with bare-integer `amount` becomes a migration later; Fake Processor because it's the **primary** test surface for all subsequent work.
**Delivers:** Monorepo layout (sibling mix projects + Release Please config), brand palette CSS vars, `Accrue.Config` (NimbleOptions), `Accrue.Error` hierarchy, `Accrue.Telemetry`, `Accrue.Money` value type, `Accrue.Processor` behaviour, `Accrue.Processor.Fake`, `Accrue.Auth` behaviour + `Auth.Default`, `Accrue.Mailer` behaviour, `Accrue.PDF` behaviour, `accrue_events` table + append-only trigger + `Events.Recorder`, skeleton `Accrue.Application` supervisor.
**Addresses:** PROC-001, PROC-003, EVT-001/002/003, OBS-001/006, AUTH-001/002, MAIL-001, PDF-001, OSS-001/002, BRND-001, BILL-090…094.
**Avoids:** Pitfalls 3 (currency), 5 (idempotency hierarchy), 8 (actor gaps), 10 (optional-dep compile).

### Phase 2: Schemas + Webhook Plumbing

**Rationale:** Schemas are pure Ecto with no processor calls; webhook pipeline has zero business logic but must be in place so Phase 3 handlers have somewhere to plug in.
**Delivers:** All Billing schemas with polymorphic owner and `data` jsonb; `Accrue.Webhook.Plug` (scoped raw-body), Verifier, `accrue_webhook_events` schema, Idempotency, Dispatcher, Worker, Handler behaviour, DefaultHandler skeleton, event-type constants module, raw-body size guard.
**Avoids:** Pitfalls 7 (webhook idempotency foundation), 8 (raw-body scoping), 9 (trace_id at ingress).

### Phase 3: Core Subscription Lifecycle

**Rationale:** Smallest slice that runs a real subscription end-to-end through Stripe test clock. **Requires Phase 0 resolved.**

### Phase Ordering Rationale

- **Order is topological, not prioritized.** Phase 1 → 2 → 3 is compile-order; nothing in Phase 3 can be written before Phase 2's schemas exist.
- **Fake Processor in Phase 1 is load-bearing.** It's not a test utility — it's the behaviour's second real implementation and the only way to unit-test Phase 3 without hitting Stripe.
- **Money type in Phase 1 is load-bearing.** Any schema built in Phase 2 with a bare integer `amount` becomes a migration later.
- **Phase 0 is the sequencing cliff.** Everything past Phase 2 stalls until the lattice_stripe Billing gap is resolved.
- **Connect sits alone in Phase 5** because it separates cleanly — no later phase depends on it.
- **Admin UI gets its own phase** because 44 IDs is the largest single surface.
- **Phase 9 is documentation + release machinery, not code.**

Phases likely needing deeper research during planning (`/gsd-research-phase` candidates):

- **Phase 3 (Core Subscription Lifecycle):** Dense with Stripe semantic gotchas. Worth a spike on subscription state-machine transition diagram and `requires_action` tagged return pattern.
- **Phase 5 (Connect):** Connect webhook endpoint conventions, destination charges vs separate charges + transfers trade-offs, capability middle states.
- **Phase 7 (Admin UI):** Mobile-first LiveView data tables are non-trivial.
- **Phase 8 (Installer):** `mix accrue.install` idempotent re-run, sigra auto-detection, migration generation — study `phx.gen.auth` and `sigra`'s installer first.

Phases with standard patterns (skip deeper research): 1, 2, 4, 6, 9.

# v1.10 prep — Metered usage surface (spike)

**Question:** What is public `Accrue.*` API vs internal / processor-only for metered billing, and what does **Fake** parity need before a **v1.10** “metered billing” milestone?

## Public entry points (host-facing)

- `Accrue.Billing.report_usage/3` and `Accrue.Billing.report_usage!/3` — documented public facade; delegates to `Accrue.Billing.MeterEventActions.report_usage/3`.
- Options (`:value`, `:timestamp`, `:identifier`, `:operation_id`, `:payload`) validated via NimbleOptions in `MeterEventActions`.

## Internal / persistence

- `Accrue.Billing.MeterEvent` schema — `accrue_meter_events` outbox + states `pending` / `reported` / `failed`.
- `Accrue.Jobs.MeterEventsReconciler` — Oban cron on `:accrue_meters` queue; retries stuck `pending` rows; emits `[:accrue, :ops, :meter_reporting_failed]` with `source: :reconciler`.
- Webhook path `billing.meter.error_report_triggered` / `v1.billing.meter.error_report_triggered` in `Accrue.Webhook.DefaultHandler` → `meter_reporting_failed` with `source: :webhook`.

## Processor / `lattice_stripe`

- `Accrue.Processor` behaviour includes `report_meter_event/1`.
- `Accrue.Processor.Stripe` forwards to Stripe meter API (idempotency at HTTP layer).
- `Accrue.Processor.Fake` implements `report_meter_event/1` and test helper `meter_events_for/1` for assertions.

## Fake parity — suggested acceptance tests (v1.10)

1. **Happy path:** `report_usage/3` with Fake → row reaches `reported`, processor stub receives deterministic `identifier`, ledger event recorded.
2. **Sync failure:** Force processor `{:error, _}` → row `failed`, **one** `[:accrue, :ops, :meter_reporting_failed]` with `source: :sync` (or documented actual atom), no duplicate on retry with same identifier.
3. **Crash after commit:** Pending row + reconciler picks up → success or `source: :reconciler` failure telemetry path.
4. **Webhook error report:** Inject meter error webhook → failed row + `source: :webhook` telemetry (existing tests may cover — map in milestone).

## Out of scope for this spike note

- Stripe Dashboard meter configuration UX (host still configures Stripe).
- **PROC-08** / non-Stripe processors.

# Research: Mailglass Integration (v1.29)

**Domain:** Transactional Email Framework (Mailglass)
**Context:** Accrue billing engine v1.29 milestone
**Researched:** Today
**Overall confidence:** HIGH

## Executive Summary

Integrating `mailglass` into `accrue` for the v1.29 milestone will significantly modernize the email pipeline. The transition will replace the Node-dependent MJML compilation (`mjml_eex`) and `phoenix_swoosh` with Mailglass's HEEx-native component system, built on top of standard `swoosh`. 

This migration improves developer experience by standardizing the email workflow within native Elixir and adds built-in idempotency, event tracking, and a powerful dev-preview UI via `mailglass_admin` inside `accrue_admin`. Since Mailglass is pre-1.0, it will be pulled in as a path or GitHub dependency.

## 1. Current Accrue Email Architecture

Accrue currently implements its own email abstraction layer built around Oban, Swoosh, and MJML:

- **Dependencies:** `swoosh` (~> 1.25), `phoenix_swoosh` (~> 1.2), `mjml_eex` (~> 0.13).
- **Core Abstraction:** `Accrue.Mailer` (facade) exposing `deliver/2` with a `type` atom and `assigns` map.
- **Async Delivery:** `Accrue.Workers.Mailer` handles job queueing via Oban (`:accrue_mailers` queue). It performs vital context hydration (locale, timezone, customer) and dynamic template resolution (honoring `:email_overrides`). 
- **Idempotency (Pitfall 7 Defense):** Uses Oban's `unique: [period: 60, fields: [:args, :worker]]` to prevent double-dispatch between billing actions and webhooks.
- **Templates:** 13 email types + 1 legacy alias defined under `Accrue.Emails.*` (e.g., `Accrue.Emails.Receipt`), leveraging `.mjml.eex` files stored in `priv/accrue/templates/emails/`.
- **Dynamic Attachments:** PDF invoices are generated at render-time and attached to specific types (`:invoice_finalized`, `:invoice_paid`).
- **Previews:** `mix accrue.mail.preview` handles rendering canned fixtures to the filesystem for testing/dev.

## 2. Mailglass Capabilities

From the `szTheory/mailglass` repository, Mailglass offers a "batteries-included transactional email framework for Phoenix":

- **HEEx-Native Components:** Replaces MJML with native Elixir HEEx tags (`container`, `row`, `column`, `button`, etc.) mapped to MSO/VML fallbacks.
- **Persistence & Idempotency:** An append-only event ledger (`mailglass_events`) backed by Postgres triggers, and native idempotency (`idempotency_key` unique index).
- **Tooling:** A dev preview LiveView (`mailglass_admin`) that auto-discovers mailables, renders responsive HTML/Text/Raw views, and allows live-editable assigns.
- **Ecosystem:** Composes seamlessly over existing `swoosh` adapters.

## 3. Integration Plan & Changes Needed

### A. Dependencies
- **Remove:** `mjml_eex` and `phoenix_swoosh` from `accrue/mix.exs`.
- **Add to Core:** `{:mailglass, path: "../mailglass"}` (or `{..., github: "szTheory/mailglass", branch: "main"}`).
- **Add to Admin:** `{:mailglass_admin, path: "../mailglass", only: [:dev]}` in `accrue_admin/mix.exs`.

### B. Template Rewrite (MJML -> HEEx)
The most time-intensive piece. All 13 MJML templates in `priv/accrue/templates/emails/` must be ported to `use Mailglass.Mailable` modules using Mailglass HEEx components.
- The `Accrue.Emails.*` modules will transition from `use MjmlEEx` to `use Mailglass.Mailable`.
- Dynamic assignments and text rendering logic (`render_text`) can be consolidated, as Mailglass provides a "Pure render pipeline" (HEEx -> CSS inlining -> auto-plaintext via Floki).

### C. Worker & Dispatch Reconciliation
- **Keep `Accrue.Workers.Mailer`:** Mailglass provides its own `deliver_later/1` (via Oban), but Accrue's worker is heavily customized for multi-tenant PDF generation (`maybe_attach_pdf`), localization fallback loops, and telemetry emitting. 
- **Recommendation:** Retain `Accrue.Workers.Mailer` as the Oban orchestrator. Have its `perform/1` step hydrate assigns, generate the PDF, instantiate a `Mailglass.Message.t()`, and call `Mailglass.deliver/1` synchronously inside the job. 
- **Idempotency Migration:** We can migrate from Oban's `unique: [period: 60]` logic to passing an explicit `idempotency_key` on the Mailglass message, leveraging Mailglass's database-level idempotency guards against Pitfall 7 (duplicate webhook dispatches).

### D. Admin Dev-Preview UI
- Mount `mailglass_admin` inside `accrue_admin/lib/accrue_admin/router.ex` to replace the filesystem-based `mix accrue.mail.preview`. 
- This gives engineers working on Accrue a localhost `/dev/mail` LiveView dashboard with instantaneous visual feedback, dark mode toggles, and dynamic assign editing.

### E. Installer & Migrations
- Since Mailglass requires a three-table migration (`mailglass_deliveries`, `mailglass_events`, `mailglass_suppressions`), `accrue`'s installation instructions (`mix accrue.install` or setup guides) must be updated to either execute `mix mailglass.install` automatically or instruct host applications to run the Mailglass migrations to prepare the database.

## 4. Risks & Gotchas

- **PDF Attachment Timing:** Accrue dynamically builds a ChromicPDF binary *inside* the Oban worker. Mailglass must natively support appending raw binary attachments (`Swoosh.Attachment.new({:data, binary})`) onto its message struct prior to `deliver()`.
- **Database Triggers:** Mailglass requires Postgres 14+ and uses immutable triggers. Ensure Accrue's test suites and `Ecto.Adapters.SQL.Sandbox` configuration play nicely with these triggers (this shouldn't be an issue, but requires verification).
- **API Stability:** Mailglass is strictly pre-v1.0 (currently targeting v0.1-v0.5 validation phases). Breaking API changes in `mailglass` could force mid-cycle updates to `accrue`. Tying the path/SHA tightly will mitigate this.

## 5. Phased Rollout Recommendation (v1.29)

1. **Phase 1: Foundation.** Add the dependency, mount `mailglass_admin` in `accrue_admin`, and verify the Postgres migrations.
2. **Phase 2: Template Porting.** Rewrite `Accrue.Emails.Receipt` and `Accrue.Emails.PaymentFailed` as a proof-of-concept. Verify HEEx to MSO VML fallback visually.
3. **Phase 3: Pipeline Wiring.** Hook up `Accrue.Workers.Mailer` to build and dispatch via `Mailglass.deliver/1`. Verify PDF attachment and telemetry.
4. **Phase 4: Full Port.** Convert the remaining 11 MJML templates and finalize the removal of `mjml_eex`.
